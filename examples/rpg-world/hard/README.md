# Hard ‚Äî adversarial problems: where the engine breaks

12 problems built to *break* the temporal search, each solvable-in-principle so a
failure is a genuine search limit (not an impossible goal). Generated by
`../../../benchmarks/scale/gen_hard.py`.

**As of v0.2.2, all 12 solve under default options** ‚Äî 10 directly (all
sub-100 ms) via the demand-guidance tiers, the numeric-threshold landmarks, and
the default-on goal-relevance pruning; the two big conjunctive orders
(`order-8`, `order-12`) via the **on-failure escalation ladder**, which hands
them to the decomposer automatically after the monolithic search exhausts
(~70‚Äì75 s total, plans validated). `ff --decompose` still gets them directly
(and shows the contract breakdown).

> **bread-line note.** Until v0.2.2 this instance was *unsolvable by
> construction* ‚Äî a domain bug: `bake-bread` produced `meals` directly, so the
> `bread` fluent had no producer (violating the hard-set's solvable-in-principle
> contract). The domain now bakes bread from flour; the instance is the real deep
> DAG its name promises, and it solves. Its distilled failing shape (`flour ‚â• 2`
> on a fully-featured hub) is what motivated graduating **goal-relevance
> pruning** to the default tier: dozens of goal-irrelevant unbounded accumulators
> (forage-food, gather-herbs, ‚Ä¶) drowned the complete search, and pruning to the
> goal-relevant op subspace turned a ~45 s exhaustion into a ~30 ms solve.

## How each original failure shape was retired

1. **Deep dependency DAGs** (`tech-*`) ‚Äî the numeric-threshold **landmark term** +
   the **`Numeric` demand tier** (default since v0.2) restore the gradient the
   delete-relaxation loses when contributions converge.
2. **Multi-path numeric goals** (`mint-fortune`) ‚Äî same demand tier: cumulative
   availability doesn't dither between interchangeable producers.
3. **Cyclic resource regeneration** (`farmstead-big`, `bread-line`) ‚Äî demand
   survives consumption across rounds; pruning keeps the search out of the
   irrelevant-accumulator swamp.
4. **The village shape** (`gather-build`) ‚Äî **goal-relevance pruning**, default-on
   since v0.2.2 (previously behind `FF_TDEMAND`).
5. **Big conjunctive goals** (`order-8/12`) ‚Äî the architecture's answer is the
   **decomposer**: split the order into per-part contracts and stitch a validated
   plan. `order-5` solves whole; 8+ wants splitting. Since the escalation ladder,
   the handoff is automatic ‚Äî a failed monolithic search falls through to the
   decomposer without a flag.

## What's robust

- **Deep travel is fine.** A 20-hop corridor solves ‚Äî `travel` gives the heuristic a
  clean distance gradient, so depth alone isn't the enemy.
- **Tight allocation is fine.** `log-scarcity` correctly splits 4 logs (saw 2 +
  burn 2) rather than greedily blowing them on one product.
- (From the broader suite: deep *linear* accumulation solves too ‚Äî `deepmine` is 15
  mining ops.)

The **exact thresholds** for each shape (binary-searched), plus the decomposition
ruleset for the subproblem-maker, are in [`../../BORDERS.md`](../../BORDERS.md) ‚Äî
e.g. linear accumulation solves to ~2000 ops, travel to ~100 hops.
