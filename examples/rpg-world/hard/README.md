# Hard — adversarial problems: where the engine breaks

12 problems built to *break* the temporal search, each solvable-in-principle so a
failure is a genuine search limit (not an impossible goal). **3 solve, 9 fail** —
and the failures cluster into a clean taxonomy. Generated by
`../../../benchmarks/scale/gen_hard.py`.

## Results

| problem | result | what it probes |
|---|---|---|
| `corridor-12` | ✅ makespan 11 | deep travel, 12-node corridor, no shortcuts |
| `corridor-20` | ✅ makespan 19 | deeper travel, 20-node corridor |
| `log-scarcity` | ✅ makespan 5 | tight allocation: split exactly 4 logs between planks + charcoal |
| `tech-steel` | ❌ ~18s | deep DAG: ore+coal→ingot→steel ×2 |
| `tech-bronze` | ❌ ~18s | deep DAG: copper+tin→ingots→bronze ×2 |
| `tech-bronze-steel` | ❌ ~17s | two deep DAGs at once |
| `bread-line` | ❌ | farm→grind→bake (fails even at `flour`, though `grain` alone solves) |
| `mint-fortune` | ❌ ~20s | coin 15 via three interchangeable paths (sell/mint/haul) |
| `order-8` | ❌ ~8s | 8-part conjunctive order |
| `order-12` | ❌ ~10s | 12-part conjunctive order |
| `farmstead-big` | ❌ ~9s | cyclic: 5 crop cycles on one field |
| `gather-build` | ❌ ~3s | the village shape (small): travel→mine→mason→build |

## What's robust (the surprises)

- **Deep travel is fine.** A 20-hop corridor solves — `travel` gives the heuristic a
  clean distance gradient, so depth alone isn't the enemy.
- **Tight allocation is fine.** `log-scarcity` correctly splits 4 logs (saw 2 +
  burn 2) rather than greedily blowing them on one product.
- (From the broader suite: deep *linear* accumulation solves too — `deepmine` is 15
  mining ops.)

## The failure taxonomy

Five shapes, and they share one root cause — **the delete-relaxed heuristic goes
flat or misleading**, so greedy search has no gradient to follow:

1. **Deep dependency DAGs** — `tech-steel`/`tech-bronze`: mine several distinct
   inputs, smelt each, then *converge* them. The relaxation under-counts the
   convergence, so guidance collapses.
2. **Big conjunctive goals** — `order-8`/`order-12`: many simultaneous
   deliverables. The boundary is real: `order-5` solves, `order-8` doesn't.
3. **Multi-path numeric goals** — `mint-fortune` (and `trade-market`/`bazaar`): a
   goal with several interchangeable producers; the search dithers between paths.
4. **Cyclic resource regeneration** — `farmstead-big`, `bread-line`: re-using one
   field across cycles toggles `tilled`/`planted`/`irrigated`, confusing both the
   heuristic and the visited-set.
5. **The village shape** — `gather-build`, `found-village`: travel + gather +
   process + build combined in one monolithic plan.

## So what

All five are either **decomposition** cases (the architecture: a scheduler slices
the goal into short contracts — and every one of these *does* solve when split) or
they land on the **temporal-heuristic** work in #45 (a landmark / non-delete-
relaxation heuristic would restore the gradient on DAGs, conjunctions, and cycles).
None are correctness bugs — every failure is a *coverage* limit on a
solvable-in-principle problem, which is exactly the map we wanted.
