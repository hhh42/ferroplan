# Village stress suite — where the monolithic plan "seems bad"

Realistic "build a whole village" scenarios for `rpg-world`, generated by
`../gen_village.py M_AGENTS HOUSES PRESTOCK SPREAD EXTRAS`. They deliberately ask
for a *full* outcome (square + houses + well/walls/feast) in **one** plan, with
many craftsmen — to find the failure cliff.

## Results (M4, contended; 90s cap)

| scenario | what | result |
|---|---|---|
| `t0-build-floor` | 1 agent, **processed** materials seeded, build square (2 houses) | **makespan 11** ✅ |
| `t1-square-pre-hub` | 2 agents, raw seeded, 2 houses+square, no travel | NO PLAN (7s) |
| `t2-square-raw-hub` | 3 agents, gather from scratch, 2 houses+square | NO PLAN (14s) |
| `t3-pre-spread` | 3 agents, raw seeded, +well+feast, travel | NO PLAN (9s) |
| `t4-raw-hub-big` | 4 agents, 3 houses+square+well, from raw | NO PLAN (20s) |
| `t5-fullday-spread` | 6 agents, 3 houses+square+well+walls+feast, travel | NO PLAN (18s) |
| `t6-everyone` | 8 agents, 4 houses + everything + coin, travel | NO PLAN (24s) |

## What it shows (the honest envelope)

1. **The build step alone is fine** (`t0`, 11): the temporal search comfortably
   handles ~10–15-action plans.
2. **Processing + building a village in one plan exceeds it.** A village goal hides
   a long chain — saw planks, mason blocks, fire bricks, *then* build each of
   several structures ≈ 20+ sequential actions — past the decision-epoch search's
   reach. (Same `#45` temporal-search-depth limit, now on a realistic shape.)
3. **More craftsmen make it WORSE, not better.** Same goal at 1 / 2 / 4 agents:
   11s → 17s → 25s, all NO PLAN. The monolithic search doesn't exploit parallelism;
   each extra agent just multiplies the branching it has to wade through.

## The takeaway (and it's the architecture, not a bug)

You never plan a whole village in one shot. The scheduler slices it into
**contract-sized sub-tasks** — "saw 12 planks", "mason 12 blocks", "raise house s0"
— each a short plan a single craftsman solves fast (see `examples/rpg-world/
contracts/`), and assigns them across the workforce. The shared stockpile carries
state between contracts. That's load-bearing: it's the only way the full village
gets built, and it's also how parallelism actually happens (N contracts on N
craftsmen), since the planner itself won't parallelize a monolithic goal.

Engine levers that raise the per-contract ceiling: a location/numeric-aware
temporal heuristic + helpful-action pruning (`#45`), and the faster derived-closure
(`#75`) for bigger maps.
